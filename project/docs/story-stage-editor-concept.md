# NarraLeaf Studio - Story Stage Editor concept

本文记录剧情编辑器的产品路线构想，不描述已经存在的实现接口，也不把还没有落地的模块写成既定架构。

## 核心判断

剧情编辑器不应该从传统脚本编辑器、纯文本语法、行式动作列表、或手工节点图开始。更合适的形态是左侧 action 行预览编辑器、右侧当前行舞台快照。

左侧不再是代码编辑器，也不再把 slash syntax 当成用户需要学习和维护的脚本文本。它应该是一个由 Studio 自己渲染的 block editor：每一行都是一个结构化 action block 的可读预览。用户看到的是每个 action 的简略信息、行号、缩进、折叠、诊断和拖拽控制；用户复制时得到的是 raw action，粘贴时恢复同样的 action。换句话说，编辑器不是把文本解析成 action，而是把 action 渲染成可编辑的行。

右侧预览显示当前 active 行执行后的状态快照。用户点哪一行，Studio 就把该 Scene 从入口状态推导到这一行，并显示当前背景、角色、表情、站位、镜头、BGM、对话框和选项状态。预览不是整个游戏运行结果，而是当前剧情位置的可解释快照。

Scene 是最小状态累加边界。跨 Scene 的变量和 flag 通过 Scene 入口快照进入；Scene 内部从开头累加到 active 行。这样避免从项目开头无限回放，也让单个 Scene 可以被独立调试、审校和演出。

## 左侧 Action 行编辑器

剧情编辑器表面上应该像普通写作工具一样顺手，但它的单位不是字符流，而是行级 block。每个 block 都有稳定 id、kind、payload、可翻译 text segment、children、diagnostics 和可选折叠状态。行号只是当前可见顺序的显示结果，不是稳定身份；翻译、配音、审校和粘贴恢复依赖 block id / textId / raw action。

一段 Scene 在编辑器里可以像这样阅读：

```text
1   [背景] 教室早晨
2   [BGM] calm_theme
3   [旁白] 今天的教室比平时更安静。
4   [Alice] 今天也要一起努力。
5   [角色] Bob 入场 · 右侧 · 400ms
6   [Bob] 可是时间不多了。
7 ▾ [选择] 接下来怎么办？
8     [选项] 留下来帮忙
9       [变量] helped_alice = true
10      [Alice] 谢谢你。
11    [选项] 先去找老师
12      [跳转] teacher_room
13  + 点击或输入以添加新行...
```

这不是文本语法示例，而是 action block 的预览形态。用户不能通过手写 `/background classroom_day` 来制造语法；用户通过输入触发器、选择补全项、编辑浮窗或 inline editor 修改结构化 action。纯文本可以作为导入导出格式，但不应成为 Studio 内部真相，也不应成为主编辑体验。

## 输入模型

编辑器底部固定存在一个输入行，用于追加新 block。点击空白输入行、按 Enter、或在当前 active 行后触发插入，都应该进入同一套输入模型。这个输入行不是数据结构上的空 block；只有用户确认输入后，才创建真实 block。

直接输入普通文本时，默认创建旁白 block。旁白创建后，行内显示旁白预览；双击旁白内容进入 inline text edit，保存时只更新对应 `StoryTextSegment.value`，保留稳定 `textId`。

输入 `/` 时展开 action 补全器。补全器列出背景、角色、镜头、音频、变量、条件、选择、跳转、等待、备注、代码块等可插入 action。选择补全项后，当前待插入行的 action 类型被确定，并立即展开 action 编辑浮窗，让用户填写必要字段，例如动画属性、角色引用、资源引用、变量值、跳转目标 Scene、condition branch 等。按 Esc 关闭 action 编辑浮窗，已确认的字段保留，未完成的必填字段通过诊断提示。

输入 `#` 时展开角色选择器。选择角色后创建 dialogue block，并在同一行角色标签后提供台词输入区域。双击角色标签重新打开角色选择器；双击台词内容进入 inline text edit。角色显示名只是派生展示，存储必须引用角色 id。

对于 action、control、jump、code、note 等非普通文本 block，双击默认打开对应编辑浮窗或 inspector。对于行内可编辑字段，例如旁白、台词、选项文本、备注，可以双击具体内容进入 inline edit。编辑器应尽量让常用写作路径留在行内，把资源选择、动画参数、条件表达式和高级字段放进轻量浮窗。

## 行级交互

每一行左侧显示行号。行号表示当前可见顺序，折叠、筛选或搜索视图变化时可以变化，不可作为生产流程身份。行号左侧用于显示结构控制，例如可折叠 control、choice、choice option、condition branch、code block 的折叠箭头。

行号右侧显示抓取 trigger。用户可以拖动 trigger 改变行位置。拖拽排序必须改变 block tree 的 parent / before 关系，而不是直接改一段文本。拖入缩进结构时必须遵守合法 parent 规则：例如 control、choice、choice option 可以承载 children；jump 是 scene transition boundary，不应拥有 children。

鼠标按下并移动可以多选行。多选应支持连续选择、Shift 扩展选择、Ctrl/Cmd 点选增减选择，并支持快捷键批量操作：复制、剪切、粘贴、删除、缩进、反缩进、折叠、展开、移动到上方/下方。多选复制得到 raw action 集合；粘贴时按原有 block tree 结构恢复。

粘贴默认插入到 active 行下方，而不是替换 active 行。若当前存在多选，粘贴行为需要明确：建议默认替换选区，或在选区最后一行下方插入，但不能让用户无法预测。这里需要在实现前定案。

按 Enter 的默认语义是准备在 active 行下方创建新 block；如果 active 行是可接收 children 的结构 block，并且当前上下文明确要求进入子内容，则可以创建 child。否则 Enter 不应靠猜测破坏结构。

Tab / Shift+Tab 的语义是调整 block 层级。缩进不是文本空格，而是 block tree 结构变更。非法缩进应禁用、回退到合法目标，或给出轻量提示。

## Control、缩进和折叠

control 类 block、choice、choice option、condition branch、code block 等会影响层级结构。它们在行号左侧显示折叠控制，折叠后隐藏 children，但保留父行的诊断摘要，例如“包含 4 行”、“2 个缺失资源”、“1 个无效跳转”。

缩进应该表达结构关系，而不是视觉装饰。choice 的直接 children 默认应是 choice option；choice option 的 children 表示选中该选项后执行的内容。condition 的直接 children 默认应是 condition branch；condition branch 的 children 表示该条件成立后执行的内容。编辑器可以在插入和拖拽时自动推荐合法结构，但不能静默创建无效结构。

折叠状态属于 editor-local 或 workspace session 状态，不应写入 StoryDocument，除非后续明确设计为团队共享视图状态。生产身份仍然来自 block id 和 textId。

## Raw Action 复制和粘贴

复制一行或多行时，剪贴板应包含 Studio 私有 MIME payload，用来保存完整 raw action、block tree、children、textId、资源引用和必要 metadata。同时可以提供普通文本 fallback，方便粘贴到外部工具阅读。

在 Studio 内粘贴私有 payload 时，应尽量恢复相同 action。需要注意两类身份：

- block id 通常应重新生成，避免在同一 scene 内产生重复 id。
- textId 是否保留需要按语义决定。普通复制粘贴如果表示“复制一份新台词”，应该生成新的 textId；如果是剪切移动或跨位置移动同一行，则应保留 textId。导入或高级粘贴可以提供“保留原翻译身份”的明确选项。

粘贴外部纯文本时，编辑器可以按行创建旁白，或识别简单的角色对话格式作为辅助导入。但这只是兼容入口，不应重新引入需要用户维护的脚本语法。

## Action 编辑浮窗

action 编辑浮窗是选择 action 类型后的主要属性编辑入口。它应该跟随当前行显示，默认轻量，不遮挡整段文档。浮窗负责资源 picker、角色 picker、Scene target picker、变量和值编辑器、动画属性、transition、duration、easing、condition expression、code language/source 等字段。

浮窗的目标是让用户完成结构化 action，而不是让用户编辑 raw JSON。常用字段优先展示，高级字段可折叠。必填字段缺失时，行内应显示诊断状态，浮窗内给出直接修复入口。

Esc 关闭浮窗；Enter 在文本字段内提交或换行的语义需要按字段定义；点击其他行应保存当前草稿或进入明确的 pending state。未完成 action 不应静默变成有效运行行为。

## 右侧 Active Snapshot

右侧预览的默认语义是 active 行的 after state，也就是执行当前行后的画面。动画和演出块还需要能切换 before / after / playback，因为用户经常要确认动作从哪里开始、到哪里结束、播放是否自然。

右侧舞台不是只读预览。用户可以拖动角色、调整缩放、换表情、移动镜头、微调透明度或选择 motion preset。关键规则是：舞台操作必须回写到当前 action block，或创建一个新的演出 block，不能直接修改累计后的快照。否则用户看到的是改了画面，但剧情文档无法解释这个变化从何而来。

如果 active 行在分支内部，block path 本身就提供了一部分分支答案。例如 active 行位于某个 choice 的第二个 option 内部，预览可以自然推导“当前预览选择了第二个选项”。这让分支预览变得直观，不需要用户额外在预览面板里重复选择。

但 active 行只能解决“路径上已经明确的分支”。外部 flag、好感度、随机结果、历史选择、持久变量仍然需要 Preview Context 来说明。遇到无法确定的状态，编辑器不应该猜，而应该把它显示为需要选择或覆盖的预览条件。

## Preview Context

Preview Context 是右侧预览使用的临时上下文。它不等同于项目数据，也不等同于真实存档。它用于回答“如果以这些变量值进入当前 Scene，active 行会是什么状态”。

它至少需要覆盖这些情况：

- 跨 Scene 的初始变量、flag、计数器和路线状态。
- 当前 Scene 的临时变量默认值。
- 当前 active 行路径之外的条件判断结果。
- 随机分支的固定结果或固定种子。
- 预览时选择的 choice 结果。
- 编辑器为了调试而临时启用的变量覆盖。

剧情编辑器应该允许用户编辑当前 Scene 的初始变量值，也应该能查看和编辑全局定义的默认变量值。这里的重点是可见性：用户需要知道一个 Scene 依赖哪些变量、哪些变量有默认值、哪些变量来自预览覆盖。

## Preview Start using 下拉

Preview 按钮右侧应该有一个下拉框，包含一个“Start using”动作，用来临时选择或覆盖当前预览使用的变量集合。

它的语义不是“修改项目默认值”，而是“当前这个预览入口从这些值开始”。例如用户可以把 `has_key` 临时设为 true，把 `alice_affection` 设为 7，然后从当前 active 行或当前 Scene 开始预览。

这个覆盖值应该对该按钮保存。也就是说，用户下次打开同一个 Scene 或同一类 Preview 入口时，之前为这个入口设置过的覆盖仍然可用。

变量定义发生变化时，下拉内容要同步反馈：

- 新增变量：出现在可覆盖列表中，使用默认值，未被用户覆盖。
- 删除变量：从可见覆盖列表移除，但不清空其他仍然有效的覆盖值。
- 重命名变量：如果系统能够识别为重命名，应迁移覆盖；不能识别时按删除旧变量、新增新变量处理。
- 类型变化：保留旧覆盖记录但标记为不兼容，等待用户确认或重设。
- 默认值变化：没有被覆盖的变量跟随新默认值；已被覆盖的变量继续使用覆盖值。

这套行为的目标是让 Preview Context 稳定、可复用、可解释，同时避免用户因为变量表调整而丢失 unrelated 的调试配置。

## 状态和分支的处理原则

Scene 内的状态推导应该是可重放的。预览某一行时，编辑器从 Scene 入口状态开始，按 block tree 的可解释顺序应用 active 行之前的 block，再应用 active 行本身。分支内 active 行可以自动提供路径答案；路径之外的不确定状态由 Preview Context 提供。

随机不能直接依赖不可复现的运行时随机数。只要随机会影响预览结果，就应该在预览中固定为一个可见结果，或固定为一个可重放的种子。用户应该能看到“这次预览采用的是随机结果 A”，并切换到其他结果。

条件隐藏和禁用不能静默吞掉。选项因为条件不可见、被禁用、或条件未知时，预览应该能解释原因。普通用户不需要看表达式细节，但需要知道“为什么这个选项没有出现”。

跳转到其他 Scene 不应该让当前 Scene 的快照推导无限延伸。当前 Scene 只负责计算到跳转 block；进入目标 Scene 后，以目标 Scene 的入口快照继续。循环、回跳和重复执行需要有明显的预览保护，避免一个 active 行无法稳定计算。

副作用要分级。设置变量、切换背景、角色移动、显示选择可以在预览中真实模拟；存档、成就、外部插件、文件写入、网络请求、真实时间等不应该在编辑器快照里真实执行，只能 dry-run 或显示占位结果。

## 值得包含的模块

Action 行编辑器：负责自绘 block 行、输入行、slash action 补全器、角色选择器、inline text edit、action 编辑浮窗、缩进、折叠、快捷键、多选、拖拽排序、查找替换和批注。它的关键不是像代码编辑器一样强，而是让用户在不学习脚本语法的情况下编辑结构化剧情。

Active Stage Snapshot：负责右侧状态快照、before / after / playback、舞台直接操作、角色和镜头可视化编辑、动画 block 回写。

Preview Context 面板：负责 Scene 初始变量、全局默认变量、Start using 覆盖、随机结果、choice trace、条件未知提示。它应该轻量但常驻可见，因为很多预览问题本质上是上下文问题。

Scene Outline：负责章节、Scene、分支摘要、入口状态、结局和跳转关系。它不需要变成主编辑器，但需要帮助用户理解当前文档在项目里的位置。

Flow Map：从剧情文档自动生成，不要求用户手动画所有节点。它用于检查路线、死路、未连接跳转、不可达 Scene、循环和结局覆盖。

Data Manager：负责角色、表情、背景、BGM、变量、flag、标签、翻译、语音状态、资源引用、批量替换和 CSV 工作流。剧情编辑器再直观，也不能牺牲批量生产能力。

Diagnostics：负责提示缺资源、缺翻译、无效变量、未知条件、随机不可复现、跳转目标不存在、分支不可达、循环风险、代码块错误、以及 active snapshot 与真实运行语义可能不一致的地方。

Dev Preview：负责真实运行验证。Active Snapshot 是编辑器推导，Dev Preview 是运行时真相。两者应该靠近，但不能假装完全等同。

## 生产流程注意点

翻译和配音需要稳定文本身份。即使用户看到的是 action 预览行，每个可翻译台词、选择文本、旁白和提示都需要稳定 `textId`，否则 CSV 往返、语音文件匹配、审校状态都会脆弱。行号不能承担这个身份。

代码块会打破“普通用户可理解”的连续性，所以需要强边界：可折叠、可标记高级、可诊断、可在大纲里弱化显示。代码块可以处理复杂逻辑，但不应该成为实现常规背景、立绘、选择、条件的唯一方式。

舞台拖拽要有清晰的写入目标。选中台词行时拖角色，可能应该提示创建 motion block；选中 motion block 时拖角色，则修改该 block。否则用户很难理解为什么某次拖动影响了哪一行。

分支内编辑要保护 block tree。普通用户可以通过 Enter、Tab 和拖拽操作分支，但编辑器必须防止把 choice option、condition branch、code block 内容破坏成无效结构。

批量管理不能被“直观编辑”挤掉。行业生产会需要按角色筛台词、按状态查未配音、按变量查引用、按资源查使用、按 Scene 导出翻译、按 tag 分配审校任务。这些都应该和剧情文档使用同一份结构化数据。

用户需要相信预览。每次快照都应该能解释“这个状态从哪里来”：入口变量、应用过的 block、当前分支路径、随机结果、覆盖值、以及被跳过的 block。解释可以折叠，但不能不存在。

## 已定交互决策

多选后粘贴插入到选区最后一行下方，而不是替换选区。没有选区时，粘贴插入到 active 行下方。

复制普通文本行后再粘贴，应生成新的 `textId`。直观性优先：用户看到的是复制了一份新文本，就应该拥有新的文本身份；没有被复制、剪切、改写或重新创建的原文本才保留原 `textId`。

`/` 只有出现在行的第一个字符时才触发 action 创建流程。如果用户没有完成 action 选择器，无论是输入了一些内容后按 Esc，还是直接按 Esc，都把当前输入作为普通旁白保留，用户可以继续按旁白编辑它。

新行输入槽不是永久数据行。它会在已有输入被使用后出现：选择 action 并展开 action 编辑浮窗后出现下一条新行；旁白在退出编辑框或手动 Enter 后出现下一条新行；Enter 会自动聚焦新行。如果用户不在最后一行按 Enter，新行应出现在当前行下方。

外部纯文本粘贴默认需要识别角色-台词格式，并尽量创建 dialogue。使用 Ctrl+Shift+V 时执行纯文本粘贴：按非空行创建旁白，跳过空行，并且不会识别命令或角色台词格式。

## 当前路线的一句话

NarraLeaf Studio 的剧情编辑器应该用结构化 action 作为真相，用自绘行预览替代脚本文本语法；像写文档一样输入旁白和台词，像导演台一样调整舞台，像调试器一样解释当前状态，像数据库一样支撑批量生产。左侧 Action 行编辑器和右侧 Active Stage Snapshot 是主形态；Scene 入口快照、Preview Context 和 Start using 覆盖是让复杂状态可控的关键。
