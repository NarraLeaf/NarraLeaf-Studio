# NarraLeaf Studio - Story Stage Editor concept

本文记录剧情编辑器的产品路线构想，不描述已经存在的实现接口，也不把还没有落地的模块写成既定架构。

## 核心判断

剧情编辑器不应该从传统脚本编辑器、行式动作列表、或手工节点图开始。更合适的形态是左侧剧情文档、右侧当前行舞台快照。

左侧是一个接近 WordPress / Monaco 混合体验的富文本剧情编辑器。用户按 Enter 添加新行，输入斜杠插入命令，普通台词保持可读文本形态。命令不是藏在复杂面板里的内部结构，而是文档中的块：背景、角色入场、镜头、选择、条件、跳转、变量修改、代码块等都可以作为块存在。代码块由指令插入，天然影响行高，并支持折叠。

右侧预览显示当前 active 行执行后的状态快照。用户点哪一行，Studio 就把该 Scene 从入口状态推导到这一行，并显示当前背景、角色、表情、站位、镜头、BGM、对话框和选项状态。预览不是整个游戏运行结果，而是当前剧情位置的可解释快照。

Scene 是最小状态累加边界。跨 Scene 的变量和 flag 通过 Scene 入口快照进入；Scene 内部从开头累加到 active 行。这样避免从项目开头无限回放，也让单个 Scene 可以被独立调试、审校和演出。

## 左侧文档体验

剧情文档表面上应该像普通写作工具，而不是要求用户先学习一门语言。

一段 Scene 可以像这样阅读：

```text
# 教室早晨

/background classroom_day
/bgm calm_theme

Alice: 今天也要一起努力。
/enter Bob from right duration 400

Bob: 可是时间不多了。

/choice
  留下来帮忙
    /set helped_alice true
    Alice: 谢谢你。
  先去找老师
    /jump teacher_room
```

它看起来像文本，但编辑器内部必须把每一行当成可识别的块。这样才能支持折叠、拖动排序、诊断、翻译导出、批量替换、右侧预览、以及舞台拖拽回写。纯文本可以作为导入导出格式，但不应成为 Studio 内部唯一真相。

斜杠菜单应该服务普通用户：插入背景、显示角色、移动角色、播放音乐、显示选择、设置变量、判断条件、跳转场景、插入备注、插入代码块。高级代码块应该明确是逃生舱，不应该成为常规剧情编写的主路径。

## 右侧 Active Snapshot

右侧预览的默认语义是 active 行的 after state，也就是执行当前行后的画面。动画和演出块还需要能切换 before / after / playback，因为用户经常要确认动作从哪里开始、到哪里结束、播放是否自然。

右侧舞台不是只读预览。用户可以拖动角色、调整缩放、换表情、移动镜头、微调透明度或选择 motion preset。关键规则是：舞台操作必须回写到当前剧情块，或创建一个新的演出块，不能直接修改累计后的快照。否则用户看到的是改了画面，但剧情文档无法解释这个变化从何而来。

如果 active 行在分支内部，文档路径本身就提供了一部分分支答案。例如光标位于某个 choice 的第二个选项内部，预览可以自然推导“当前预览选择了第二个选项”。这让分支预览变得直观，不需要用户额外在预览面板里重复选择。

但 active 行只能解决“路径上已经明确的分支”。外部 flag、好感度、随机结果、历史选择、持久变量仍然需要 Preview Context 来说明。遇到无法确定的状态，编辑器不应该猜，而应该把它显示为需要选择或覆盖的预览条件。

## Preview Context

Preview Context 是右侧预览使用的临时上下文。它不等同于项目数据，也不等同于真实存档。它用于回答“如果以这些变量值进入当前 Scene，active 行会是什么状态”。

它至少需要覆盖这些情况：

- 跨 Scene 的初始变量、flag、计数器和路线状态。
- 当前 Scene 的临时变量默认值。
- 当前 active 行路径之外的条件判断结果。
- 随机分支的固定结果或固定种子。
- 预览时选择的 choice 结果。
- 编辑器为了调试而临时启用的变量覆盖。

剧情编辑器应该允许用户编辑当前 Scene 的初始变量值，也应该能查看和编辑全局定义的默认变量值。这里的重点是可见性：用户需要知道一个 Scene 依赖哪些变量、哪些变量有默认值、哪些变量来自预览覆盖。

## Preview Start using 下拉

Preview 按钮右侧应该有一个下拉框，包含一个“Start using”动作，用来临时选择或覆盖当前预览使用的变量集合。

它的语义不是“修改项目默认值”，而是“当前这个预览入口从这些值开始”。例如用户可以把 `has_key` 临时设为 true，把 `alice_affection` 设为 7，然后从当前 active 行或当前 Scene 开始预览。

这个覆盖值应该对该按钮保存。也就是说，用户下次打开同一个 Scene 或同一类 Preview 入口时，之前为这个入口设置过的覆盖仍然可用。

变量定义发生变化时，下拉内容要同步反馈：

- 新增变量：出现在可覆盖列表中，使用默认值，未被用户覆盖。
- 删除变量：从可见覆盖列表移除，但不清空其他仍然有效的覆盖值。
- 重命名变量：如果系统能够识别为重命名，应迁移覆盖；不能识别时按删除旧变量、新增新变量处理。
- 类型变化：保留旧覆盖记录但标记为不兼容，等待用户确认或重设。
- 默认值变化：没有被覆盖的变量跟随新默认值；已被覆盖的变量继续使用覆盖值。

这套行为的目标是让 Preview Context 稳定、可复用、可解释，同时避免用户因为变量表调整而丢失 unrelated 的调试配置。

## 状态和分支的处理原则

Scene 内的状态推导应该是可重放的。预览某一行时，编辑器从 Scene 入口状态开始，按文档顺序应用 active 行之前的块，再应用 active 行本身。分支内 active 行可以自动提供路径答案；路径之外的不确定状态由 Preview Context 提供。

随机不能直接依赖不可复现的运行时随机数。只要随机会影响预览结果，就应该在预览中固定为一个可见结果，或固定为一个可重放的种子。用户应该能看到“这次预览采用的是随机结果 A”，并切换到其他结果。

条件隐藏和禁用不能静默吞掉。选项因为条件不可见、被禁用、或条件未知时，预览应该能解释原因。普通用户不需要看表达式细节，但需要知道“为什么这个选项没有出现”。

跳转到其他 Scene 不应该让当前 Scene 的快照推导无限延伸。当前 Scene 只负责计算到跳转块；进入目标 Scene 后，以目标 Scene 的入口快照继续。循环、回跳和重复执行需要有明显的预览保护，避免一个 active 行无法稳定计算。

副作用要分级。设置变量、切换背景、角色移动、显示选择可以在预览中真实模拟；存档、成就、外部插件、文件写入、网络请求、真实时间等不应该在编辑器快照里真实执行，只能 dry-run 或显示占位结果。

## 值得包含的模块

剧情文档编辑器：负责块级编辑、斜杠菜单、缩进、折叠、代码块、快捷键、拖拽排序、查找替换和批注。它的关键不是像代码编辑器一样强，而是让普通用户写起来没有阻力。

Active Stage Snapshot：负责右侧状态快照、before / after / playback、舞台直接操作、角色和镜头可视化编辑、动画块回写。

Preview Context 面板：负责 Scene 初始变量、全局默认变量、Start using 覆盖、随机结果、choice trace、条件未知提示。它应该轻量但常驻可见，因为很多预览问题本质上是上下文问题。

Scene Outline：负责章节、Scene、分支摘要、入口状态、结局和跳转关系。它不需要变成主编辑器，但需要帮助用户理解当前文档在项目里的位置。

Flow Map：从剧情文档自动生成，不要求用户手动画所有节点。它用于检查路线、死路、未连接跳转、不可达 Scene、循环和结局覆盖。

Data Manager：负责角色、表情、背景、BGM、变量、flag、标签、翻译、语音状态、资源引用、批量替换和 CSV 工作流。剧情编辑器再直观，也不能牺牲批量生产能力。

Diagnostics：负责提示缺资源、缺翻译、无效变量、未知条件、随机不可复现、跳转目标不存在、分支不可达、循环风险、代码块错误、以及 active snapshot 与真实运行语义可能不一致的地方。

Dev Preview：负责真实运行验证。Active Snapshot 是编辑器推导，Dev Preview 是运行时真相。两者应该靠近，但不能假装完全等同。

## 生产流程注意点

翻译和配音需要稳定行 ID。即使用户看到的是富文本，每个可翻译台词、选择文本、旁白和提示都需要稳定身份，否则 CSV 往返、语音文件匹配、审校状态都会脆弱。

代码块会打破“普通用户可理解”的连续性，所以需要强边界：可折叠、可标记高级、可诊断、可在大纲里弱化显示。代码块可以处理复杂逻辑，但不应该成为实现常规背景、立绘、选择、条件的唯一方式。

舞台拖拽要有清晰的写入目标。选中台词行时拖角色，可能应该提示创建 motion block；选中 motion block 时拖角色，则修改该 block。否则用户很难理解为什么某次拖动影响了哪一行。

分支内编辑要保护缩进结构。普通用户可以通过 Enter 和 Tab 操作分支，但编辑器必须防止把 choice 子项、condition 子项、代码块内容破坏成无效结构。

批量管理不能被“直观编辑”挤掉。行业生产会需要按角色筛台词、按状态查未配音、按变量查引用、按资源查使用、按 Scene 导出翻译、按 tag 分配审校任务。这些都应该和剧情文档使用同一份结构化数据。

用户需要相信预览。每次快照都应该能解释“这个状态从哪里来”：入口变量、应用过的块、当前分支路径、随机结果、覆盖值、以及被跳过的块。解释可以折叠，但不能不存在。

## 当前路线的一句话

NarraLeaf Studio 的剧情编辑器应该像写文档一样创作剧情，像导演台一样调整舞台，像调试器一样解释当前状态，像数据库一样支撑批量生产。左侧 Story Document Editor 和右侧 Active Stage Snapshot 是主形态；Scene 入口快照、Preview Context 和 Start using 覆盖是让复杂状态可控的关键。
