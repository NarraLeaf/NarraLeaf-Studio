# Visual Editor M2-B + Visual Editor M3 实现方案

## Overview

前置默认成立：`Visual Editor M1`、`Visual Editor M2-A`、`Blueprint System M2`、`Blueprint System M3-min`、`Visual Editor M4-lite` 已完成。

**实施状态（2026-04-03）**：已在主工程内编码落地（共享流式布局、`nl.stack` / `nl.scroll` / `nl.spacerDivider` / `nl.listRepeater`、Surface 列表 Link 摘要、三份 visual-editor 文档与 `project/examples/visual-editor/` 示例）。原计划的独立文件 `project/docs/implementation-plans/p3-ve-m2b-ve-m3-plan.md` 未创建；实施细节以本 plan 的 Implementation Units 与代码为准。

## 已确认的规划决策

- `Option List / Repeater` 采用 **同一类最小 List/Repeater widget**：当前只做结构与设计时预览，`Option List` 作为官方组合范式，而不是第二套主 widget。
- `Visual Editor M3` 的官方组合范式采用 **文档 + 示例 Surface/示例资产**，不做编辑器内 starter 入口。
- 示例资产放在独立目录：`project/examples/visual-editor/...`。
- `Spacer / Divider` 采用 **一个共享 widget + mode 切换**。
- `M2-B` 补齐时同时引入 **共享布局模型**，把“根节点自由定位 / 布局容器内流式排列”从目标方向落成显式规则，而不是长期停留在 widget 内部特判。
- 本期默认最小边界：
- `Stack`：方向、gap、主轴与交叉轴对齐、内边距。
- `Scroll`：视口、单轴滚动模式、裁剪。
- `Spacer / Divider`：mode、尺寸/线厚、颜色、inset。
- `List / Repeater`：模板子树、设计时预览数量、项间距；不做运行时数据源。

## 范围与非目标

### 本期范围

- 补齐四类剩余 widget：`Stack`、`Scroll`、`Spacer / Divider`（共享 widget）、`List / Repeater`（共享 widget）。
- 把流式布局所需的最小共享规则补进 [src/shared/types/ui-editor/document.ts](src/shared/types/ui-editor/document.ts)、[src/renderer/lib/ui-editor/runtime/EditorNodeWrapper.tsx](src/renderer/lib/ui-editor/runtime/EditorNodeWrapper.tsx) 及相关交互链路。
- 沉淀官方 VN 组合范式：至少覆盖 `Dialog`、`Choice Menu`、`Notification / Toast`、`Settings`、`Save / Load`、`Overlay / Pause Menu`。
- 明确 Surface 命名、组织、Link 与复制复用约定，并与现有 `Scene Properties` / `Surfaces Panel` 口径对齐。
- 通过现有文档与示例资产提供可复制、可 Link、可对照的官方范式。

### 非目标

- 不引入模板系统、preset 系统、组件系统、编辑器内 starter 入口。
- 不把 `List / Repeater` 扩成运行时数据源、虚拟化列表或完整模板实例化系统。
- 不把 `Option List` 做成第二套独立主 widget。
- 不进入 `Blueprint M4-full` 的绑定编辑、图编辑、属性状态机。
- 不把 `Visual Editor` 变成第二个 `Dev Mode`；编辑器仍然只做静态 / 布局预览。
- 不提前实现 `Visual Editor M5` 的校验体系，只允许在本期文档里为后续留挂点。

## 关键技术方向

### 1. 共享布局模型补齐

本期不把布局能力继续埋在 widget 特判里，而是补一个最小共享规则层：

- 根 Surface 及自由定位元素继续支持现有绝对布局语义。
- 进入 `Stack / Scroll / List / Repeater` 这类流式父节点后，子节点以“由父布局管理”的方式参与排列。
- 共享布局模型只补最小 discriminated rule，不一次扩成完整 CSS/Flex DSL。
- 交互层明确区分两类节点：
- **自由定位节点**：仍允许拖拽改 `x / y / width / height`。
- **流式子节点**：优先限制直接自由拖拽，改由父容器规则和 Inspector 驱动排列。

### 2. 剩余 widget 的最小能力边界

#### `Stack`

- 目标：成为最小 Auto Layout 容器，而不是完整 FlexBox 映射。
- 最小边界：`direction`、`gap`、`padding`、`alignItems`、`justifyContent`。
- 非目标：wrap、复杂 basis/grow/shrink 矩阵、响应式断点。

#### `Scroll`

- 目标：提供可视口裁剪的滚动容器，服务长列表、设置页、存读档页。
- 最小边界：单轴滚动模式、裁剪、内容承载、设计时滚动预览位。
- 非目标：嵌套滚动冲突优化、惯性滚动、虚拟化、平台级 scrollbar 皮肤系统。

#### `Spacer / Divider`（共享 widget）

- 目标：一个共享 layout primitive，通过 mode 切换承担“空白占位”与“可见分隔线”。
- 最小边界：`mode`、尺寸/线厚、颜色、inset、方向。
- 语义要求：在流式容器里表现最好；在自由定位场景下仍需有可解释行为，但不鼓励作为主要用法。

#### `List / Repeater`（共享 widget）

- 目标：承载重复结构和设计时预览，而不是承载真实运行时列表数据。
- 最小边界：模板子树、预览数量、项间距、基础方向；必要时允许与 `Divider` mode 协同形成列表分隔线。
- 非目标：真实数据源、item context、声明绑定 UI、运行时动态实例化。
- `Option List` 的落点：文档与示例范式，不新增独立 `OptionList` 主 widget。

### 3. Surface 命名、组织、Link 与复用约定

本期以“约定 + 示例 + 现有 Link 能力”解决复用，不引入模板系统：

- 命名约定继续围绕现有 `App Surface` / `Stage Surface` 模型，不另起 VN 专用主模型。
- Stage 复用 App UI 的唯一正式路径仍是 `link.kind = "appSurface"`。
- 组织约定以“用途 + 场景层级”表达，例如主页面、叠层页面、slot 页面，而不是以未来模板概念命名。
- 文档要明确什么时候用 `App Surface`，什么时候用 `Stage Surface`，什么时候优先 `Link` 而不是复制。
- 示例资产要直接演示“同一个 Settings/App Surface 被 Stage layer 复用”的推荐做法。

### 4. 组合范式的承载方式

- 主要承载：更新现有文档 [project/docs/visual-editor.md](project/docs/visual-editor.md)、[project/docs/visual-editor-milestones.md](project/docs/visual-editor-milestones.md)、[project/docs/visual-editor-implementation-guide.md](project/docs/visual-editor-implementation-guide.md)。
- 示例承载：新增 `project/examples/visual-editor/...` 官方示例资产目录。
- 不做：编辑器内 `New from Pattern`、Starter Gallery、模板库面板。
- 复用路径：用户通过阅读文档、打开示例资产、复制现有 Surface、使用 `Stage Surface Link` 完成迁移。

**Approach:**
- 文档负责解释范式目标、层级结构、命名、推荐 Link/复制策略。
- 示例资产负责提供可复制、可拆解、可对照的实际结构。
- 示例统一只使用本阶段可用 widget，不预设未来模板系统或运行时数据源。

**Verification:**
- 用户可以不借助 starter 入口，仅凭文档和示例就复现高频 VN 界面。
- 示例资产能清楚表达 `Option List = List / Repeater + Button/Text + 命名约定`。

## UI 一致性要求

- 新 widget 的 Inspector、Help Text、空状态与按钮风格必须沿用现有 `PropertiesPanel` / `ReadonlyBlueprintSection` / `BlueprintEntryTab` 的视觉语言，不引入新的面板体系。
- Docker 与 Insert 入口沿用 [src/renderer/lib/ui-editor/docker/UIEditorDockerBar.tsx](src/renderer/lib/ui-editor/docker/UIEditorDockerBar.tsx) 现有尺寸、圆角、边框和 hover 模式。
- `Scene Properties` 和 `Surfaces Panel` 的新增文案必须与现有字段命名保持一致，避免一处叫 `Link`、另一处叫 `Reuse`、第三处叫 `Reference`。
- 示例范式只展示本阶段真实可用的结构，不用假的可点入口误导用户。

## 验收标准

- Insert 菜单、Docker、文档创建、运行时渲染、Properties Inspector 全部能识别四类新增 widget。
- `Stack / Scroll / List / Repeater` 的子节点不再被全局绝对定位模型错误处理。
- `Option List` 能通过共享 `List / Repeater` widget + 基础 widget 组合稳定搭建，无需独立第二主控件。
- `Visual Editor M3` 的文档与示例资产能明确回答：怎么命名 Surface、怎么组织、什么时候用 Link、什么时候复制。
- `project/docs/visual-editor-milestones.md`、`project/docs/visual-editor.md`、`project/docs/visual-editor-implementation-guide.md` 对 P3 范围和边界的表述保持一致。
- 新增 UI 风格与当前 Workspace 其他部分一致。

## 风险与缓解

- **共享布局模型波及交互层**：通过最小 discriminated rule 控制范围，不一次重写所有布局语义。
- **`List / Repeater` 被误做成半套模板系统**：明确写死本期不做运行时数据源和 starter 入口。
- **共享 `Spacer / Divider` 造成用户理解成本**：通过偏 Web 命名、简化 Inspector、把主要用法限制在流式容器中缓解。
- **示例资产无人发现**：文档必须显式链接 `project/examples/visual-editor/...`，不能只创建目录不接入口。
